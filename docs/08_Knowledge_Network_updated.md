# 1. Purpose

The Knowledge Network defines how Milan Red Line guides connect to each other and how editorial authority grows over time.

Its purpose is to create an ecosystem where readers naturally move from broad orientation to increasingly specific answers without encountering duplicated content.

The Knowledge Network serves three goals:

- help visitors find the next useful answer;
- strengthen topical authority;
- keep the editorial architecture scalable.

Internal linking is an editorial tool before it is an SEO technique.

---

# 2. Core Principles

The Knowledge Network is based on five permanent principles:

- every guide has a clearly defined role;
- every guide belongs to a content pillar;
- links answer the reader's next logical question;
- content is connected, not repeated;
- navigation must feel natural rather than engineered.

A guide should never exist in isolation.

---

# 3. Topical Authority

Milan Red Line is not trying to rank a single page in isolation.

The long-term objective is to become a trusted independent resource about:

- staying in and around Milan;
- moving around Milan efficiently;
- exploring Northern Italy by train and public transport;
- making practical travel decisions before arrival.

Every new guide should strengthen one or more of these areas.

Topical authority is built through:

- coherent content clusters;
- clear guide roles;
- useful internal links;
- consistent metadata;
- editorial depth without duplication;
- progressive coverage of real visitor questions.

---

# 4. Content Pillars

The current SEO and editorial structure is organised around four macro-pillars.

## Accommodation

Purpose:

Help travellers decide where to stay.

Typical topics include:

- best areas;
- Sesto San Giovanni;
- first-time visitors;
- families;
- business travellers;
- Rho Fiera;
- event and institution-based accommodation needs.

## Transport

Purpose:

Explain how to move around Milan and how to arrive or continue beyond the city.

Typical topics include:

- public transport;
- metro;
- tickets;
- airports;
- railway stations;
- journeys without a car.

## Day Trips

Purpose:

Show how Milan can function as a base for exploring Northern Italy.

Typical topics include:

- Monza;
- Lake Como;
- Lecco;
- Bergamo;
- other destinations reachable by rail.

## Local Knowledge

Purpose:

Answer practical questions visitors often have before or during their stay.

Typical topics may include:

- supermarkets;
- restaurants;
- parking;
- safety;
- neighbourhood services;
- everyday travel tips.

A new pillar should be created only when the existing architecture can no longer represent a meaningful group of user needs.

# 5. Network Structure

```text
Pillar
   │
   ├── Hub
   │      │
   │      ├── Connector
   │      │        │
   │      │        ├── Specialist Guide
   │      │        ├── Specialist Guide
   │      │        └── Specialist Guide
```

Readers should progressively discover more detailed information.

---

# 6. Guide Roles

## Hub

A Hub provides broad orientation.

Objective:

Help the reader understand a topic and identify the most useful next step.

Typical characteristics:

- strategic overview;
- limited procedural detail;
- links towards Connector Guides;
- clear distinction between major subtopics.

A Hub should not become an encyclopedia.

## Connector

A Connector bridges broad orientation and specialist procedures.

Objective:

Explain how a system works as a whole while preparing the reader for deeper guides.

Typical characteristics:

- broad but practical scope;
- decision-oriented content;
- strong internal-linking role;
- explicit boundaries to prevent duplication.

Example:

`How to Use Milan Public Transport`

## Specialist Guide

A Specialist Guide answers one specific question in depth.

Objective:

Provide complete practical information for a narrow search intent.

Examples:

- ATM Tickets Explained;
- Malpensa to Milan;
- Linate to Milan;
- How to Use the Milan Metro.

A Specialist Guide should not repeat the strategic overview already covered by its Hub or Connector.

---

# 7. Internal Linking Rules

Every contextual link should satisfy at least one condition:

- answers the next logical question;
- reduces duplication;
- improves understanding;
- supports planning;
- connects the current guide to the correct level of the network.

Links should be distributed naturally through the article.

Avoid:

- generic SEO-only anchor text;
- repeated links to the same destination without a clear purpose;
- forced references to apartments;
- lists of unrelated guides inside the article body;
- links to pages that do not yet exist.

The preferred range is generally three to five contextual links per guide, but editorial usefulness takes precedence over a fixed number.

---

## Maintaining Editorial Clusters

### Principle: Redistribute, Don't Accumulate

Publishing a new specialist guide should improve the entire editorial cluster, not simply increase its size.

When a specialist guide is added, the surrounding Hub and Connector guides should be reviewed to ensure that information is distributed across the cluster rather than duplicated.

### Editorial Maintenance Rules

For every new specialist guide:

- review all closely related Hub and Connector guides;
- replace detailed explanations with concise summaries where appropriate;
- add natural contextual links to the new guide;
- remove duplicated concepts within the same guide;
- eliminate unnecessary repetition across the cluster;
- keep each guide focused on a single search intent.

The objective is not to make existing guides longer, but clearer and more specialised.

### Rule of Thumb

> Every new specialist guide should make the surrounding guides simpler, not longer.

A mature Knowledge Network grows by distributing knowledge across focused pages instead of accumulating the same information everywhere.

# 8. Related Guides

Related Guides complement contextual links but do not replace them.

Rules:

- normally show three guides;
- prioritise editorial relevance;
- avoid repeating the same reader journey;
- include the most natural next steps;
- update the central configuration whenever a new guide is added or a Legacy guide becomes Current.

The related-guides map must be expanded incrementally.

Whenever an older guide is migrated or updated to the latest Reading Experience standard, update its related-guides configuration within the same sprint.

---

# 9. Apartment Links and Conversion

Apartment links belong in the Knowledge Network only when they help the reader act on the information provided.

Use apartment links when:

- the reader is actively choosing where to stay;
- the location or transport connection is directly relevant;
- the guide has already answered the primary question;
- the link supports planning rather than interrupting it.

Do not insert apartment links only for SEO or conversion pressure.

The standard remains:

> Help first. Trust first. Booking second.

---

# 10. Legacy Guide Migration

Whenever a Legacy guide enters a sprint:

- review its role in the Knowledge Network;
- review contextual links;
- update Related Guides;
- verify metadata and search aliases;
- verify the Reading Experience Framework;
- remove duplicated content;
- promote the guide to Current whenever reasonably possible.

The objective is to modernise the project incrementally rather than schedule a separate migration for every older page.

---

# 11. Planning a New Guide

Before creating a new guide, answer these questions:

1. Which pillar does it belong to?
2. Does an appropriate Hub already exist?
3. Is the guide a Hub, Connector or Specialist?
4. Which existing guide should link to it?
5. Which guides should it link to?
6. Does it duplicate content already covered elsewhere?
7. Does the search intent justify a separate page?
8. Can the guide remain evergreen?

A guide should not be approved until its network role is clear.

---

# 12. Cluster Growth Strategy

New guides should strengthen existing clusters before creating new ones.

Priority order:

1. strengthen existing pillars;
2. complete Hub and Connector coverage;
3. publish high-value Specialist Guides;
4. improve incoming and outgoing links;
5. create new pillars only when justified.

The objective is not page count.

The objective is a coherent, useful and increasingly authoritative network.

---

# 13. Quality Checklist

Before publication verify:

- pillar identified;
- guide role defined;
- primary search intent preserved;
- contextual links reviewed;
- Related Guides updated;
- incoming links planned;
- duplication avoided;
- apartment links used only where relevant;
- future links not created before pages exist;
- guide strengthens the wider cluster.
