# PM Continuity Notes
## Part 1 — Project Mindset & Decision Philosophy

Status: Living document

Purpose

This document complements the permanent project documentation.

It intentionally does NOT repeat information already covered by the Blueprint, Roadmap, Development Standards or other permanent modules.

Instead, it captures the practical reasoning, strategic judgement and decision-making philosophy that emerged during the evolution of Milan Red Line.

The objective is that a future Project Manager can make decisions consistent with the project's direction even without access to previous conversations.

---

# 1. The Project Is Bigger Than Two Apartments

One of the most important mindset shifts made during the project was understanding that Milan Red Line is not fundamentally an accommodation website.

It is an information platform that happens to offer accommodation.

This distinction affects almost every strategic decision.

Apartment websites compete almost exclusively on:

- price;
- photos;
- location;
- reviews.

Editorial platforms compete on:

- usefulness;
- trust;
- authority;
- long-term visibility.

Milan Red Line deliberately chose the second path.

The apartments are the business.

The website is the long-term asset.

Every improvement should strengthen that asset.

---

# 2. Help Before Selling Is Not Marketing

"Help first. Trust first. Booking second."

This is often misunderstood as a branding slogan.

It is not.

It is the project's main decision filter.

Whenever two possible implementations exist, the preferred one is almost always the one that increases usefulness, even if it delays commercial visibility.

Examples.

Instead of:

"Book now."

Prefer:

"Here is everything you need to know."

The expectation is that trust naturally produces bookings.

This principle has repeatedly proven correct throughout the project.

---

# 3. Simplicity Is a Competitive Advantage

The project intentionally avoids unnecessary complexity.

This applies equally to:

- architecture;
- editorial structure;
- design;
- navigation;
- components;
- future features.

A feature is never considered valuable simply because it is technically interesting.

Complexity must continuously justify its existence.

Whenever two implementations solve the same problem, the simpler one should almost always win.

---

# 4. Documentation Is Part of the Product

Documentation is not considered project overhead.

It is part of the product itself.

Every major sprint should leave three things behind:

- working implementation;
- updated documentation;
- reusable knowledge.

The goal is that future work becomes progressively easier rather than progressively harder.

Good documentation reduces future development cost.

---

# 5. Local Knowledge Is the Real Differentiator

Many accommodation websites can describe Milan.

Few can explain North Milan well.

Even fewer can explain Sesto San Giovanni from the perspective of someone actually staying there.

This became one of Milan Red Line's strongest differentiators.

The objective is not to become another generic travel guide.

The objective is to become the most trustworthy source for planning a stay based around Sesto San Giovanni and the M1 Red Line.

Every Local Knowledge guide should reinforce this identity.

---

# 6. Growth Comes From Clusters, Not Isolated Pages

One of the most important lessons learned is that isolated content rarely creates long-term value.

Whenever a new guide is proposed, the first question should not be:

"Is this a good guide?"

Instead ask:

"What cluster does this strengthen?"

Strong clusters repeatedly proved more valuable than unrelated high-quality pages.

Every guide should reinforce an existing editorial ecosystem whenever reasonably possible.

---

# 7. Finish Before Expanding

Throughout development, a recurring principle emerged.

It is almost always better to finish an existing area than to start a new one.

Examples include:

- completing Reading Experience migrations;
- completing Knowledge Network links;
- updating Related Guides;
- documenting architectural decisions.

Incomplete systems generate maintenance.

Completed systems generate momentum.

This principle should continue guiding roadmap decisions.

---

# 8. Reuse Is a Product Philosophy

Reusable components are not simply a development preference.

They represent the project's broader philosophy.

The same applies to:

- editorial structures;
- design patterns;
- workflows;
- documentation.

Whenever something useful appears twice, consider whether it should become part of the permanent system.

This philosophy has significantly reduced technical debt while improving consistency across the project.

---

# 9. Every Sprint Should Leave the Project Cleaner

A sprint should not only add value.

It should reduce future work.

Examples:

- updating documentation while implementing;
- completing inbound links when publishing a guide;
- migrating Legacy guides while editing them;
- improving shared components rather than duplicating code.

The project should become progressively easier to maintain.

Never merely larger.

---

# 10. Long-Term Thinking Always Wins

Many attractive ideas have deliberately been postponed.

Not because they were bad.

Because they were early.

The project repeatedly benefited from asking:

"Is this the highest-value use of our time today?"

Very often, publishing another excellent guide produced more long-term value than implementing another feature.

Future PMs should continue making decisions through opportunity cost rather than enthusiasm.

The roadmap should remain disciplined.

Useful content generally creates more durable value than additional functionality.

# PM Continuity Notes
## Part 2 — Prioritisation & Strategic Thinking

Status: Living document

---

# 11. Always Optimise for Long-Term Value

The project should rarely optimise for the fastest visible result.

Instead, optimise for the highest cumulative value.

Many decisions become easier when asking:

> "Will this still matter in one year?"

If the answer is yes, it is usually a good investment.

Examples:

- improving documentation;
- strengthening the Knowledge Network;
- creating reusable components;
- writing evergreen guides.

These activities often appear slower initially but repeatedly proved to deliver the greatest long-term return.

---

# 12. Build Foundations Before Features

Throughout development, one recurring pattern emerged.

Whenever the project invested first in foundations, later work became dramatically faster.

Examples include:

- Reading Experience before publishing more guides;
- Knowledge Network before aggressive content expansion;
- Brand Guidelines before visual refinements;
- Guide Search architecture before UX improvements.

The preferred order is always:

Foundation

↓

Implementation

↓

Optimisation

Never reverse this sequence.

---

# 13. Avoid Solving Problems That Do Not Yet Exist

Many ideas were intentionally postponed despite being technically interesting.

Examples included:

- advanced search filters;
- AI features;
- database adoption;
- sophisticated animations;
- user accounts;
- booking engines.

The reason was simple.

The current project size did not justify their complexity.

Future PMs should remain cautious about solving hypothetical future problems.

Real bottlenecks deserve solutions.

Imagined bottlenecks usually do not.

---

# 14. Prefer Evolution Over Replacement

The project almost never replaces working systems.

Instead, it evolves them.

Examples:

Guide Search was extended.

It was not rewritten.

Reading Experience evolved incrementally.

It was never discarded.

Knowledge Network expanded cluster by cluster.

This philosophy dramatically reduced regressions and preserved development momentum.

Whenever possible:

Improve.

Do not restart.

---

# 15. Complete Editorial Loops

Publishing a guide is only one step.

A guide should normally leave the sprint with:

- outbound links;
- inbound links planned or implemented;
- Related Guides updated;
- metadata complete;
- hero image;
- documentation updated.

A published guide that remains isolated is only partially finished.

The Knowledge Network should grow together with content.

---

# 16. Use Analytics to Validate, Not to Dictate

Analytics should answer questions.

It should not create them.

Avoid publishing content simply because a keyword appears promising.

Instead:

Publish content that naturally fits the editorial strategy.

Then use Analytics and Search Console to understand:

- what users discover;
- what deserves expansion;
- what requires clarification.

The project should remain editorially driven rather than algorithmically driven.

---

# 17. Every New Guide Should Strengthen Positioning

A guide is valuable not only because it ranks.

It is valuable because it reinforces Milan Red Line's identity.

Repeatedly ask:

"What does this page teach visitors about Milan Red Line?"

If the answer could apply equally to any generic travel website, the guide probably lacks distinctive value.

The strongest guides reinforce at least one of these ideas:

- transport convenience;
- Sesto San Giovanni as a strategic base;
- authentic local knowledge;
- practical planning.

---

# 18. Positioning Evolves Gradually

One of the biggest strategic evolutions was recognising that the project should not stop at Milan.

Initially, the narrative focused almost entirely on reaching central Milan.

Later, a broader positioning emerged.

The apartments are a base for exploring:

- Milan;
- Monza;
- North Milan;
- Lake Como;
- Northern Italy by rail.

This evolution did not replace the original positioning.

It expanded it.

Future growth should continue following this principle.

Expand.

Do not contradict.

---

# 19. Editorial Depth Beats Editorial Breadth

When choosing between:

Option A

Publish five average guides.

Option B

Publish one outstanding guide and fully integrate it into the Knowledge Network.

Option B almost always wins.

Depth creates authority.

Breadth without depth creates maintenance.

Future planning should continue favouring complete editorial clusters over isolated publications.

---

# 20. Every Sprint Should Improve the System

The most successful sprints were rarely those producing the largest number of visible changes.

Instead, they improved the project's overall capability.

Examples include:

- introducing Reading Experience;
- standardising documentation;
- creating reusable components;
- implementing Guide Search;
- formalising Brand Guidelines.

These improvements continue generating value long after the sprint ends.

When evaluating future work, always ask:

"Does this sprint leave Milan Red Line stronger than before, or merely larger?"

The preferred answer is always:

Stronger.

# PM Continuity Notes
## Part 3 — Lessons Learned & Strategic Judgement

Status: Living document

---

# 21. The Best Ideas Usually Emerge During Execution

Many important improvements were not planned at the beginning of a sprint.

They emerged while solving another problem.

Examples include:

- the Reading Experience becoming a complete framework rather than a layout;
- the Knowledge Network evolving into a permanent editorial architecture;
- the guide metadata model becoming the foundation for search;
- Brand Guidelines becoming a permanent project document.

Future PMs should therefore distinguish between:

- changing direction;
- discovering a better solution.

The roadmap should provide direction, not rigidity.

---

# 22. Avoid Perfection Before Validation

The project deliberately stopped refining several features once they became "good enough".

Examples include:

- search interface;
- guide cards;
- homepage layout;
- visual effects.

The reason is simple.

Without user behaviour data, additional refinements are mostly subjective.

Once a solution is:

- coherent;
- maintainable;
- pleasant to use;

it is usually better to move on.

Future improvements should increasingly be justified by Analytics, Search Console or real user observations.

---

# 23. The Project Benefits from Clear Chat Specialisation

One of the most successful operational decisions was separating responsibilities across dedicated chats.

Instead of maintaining a single enormous conversation, responsibilities were divided between:

- Project Management;
- Development;
- Content & SEO;
- UX / Design;
- Analytics & Growth.

This improved both response quality and continuity.

The Project Manager should remain responsible for priorities and cross-domain decisions rather than implementation details.

---

# 24. Keep the Number of Open Topics Small

Whenever too many parallel initiatives were considered, decision quality decreased.

The project progresses best when only a small number of strategic objectives are active.

A typical pattern is:

one major development sprint;

one editorial sprint;

one documentation update.

This rhythm proved sustainable and kept momentum high without creating unnecessary context switching.

---

# 25. Technical Debt Is Often Editorial Debt

During the project it became clear that maintenance effort is not generated only by code.

Editorial inconsistency creates its own form of technical debt.

Examples include:

- outdated internal links;
- inconsistent metadata;
- Legacy guides;
- incomplete Related Guides;
- duplicated explanations.

Future PMs should treat editorial maintenance with the same importance as code maintenance.

---

# 26. Documentation Should Capture Decisions, Not Conversations

A recurring lesson was that conversations are poor long-term storage.

What matters is not how a decision was reached.

What matters is:

- the decision;
- the reasoning behind it;
- the consequences.

Whenever an idea becomes stable, it should migrate into documentation.

Future PMs should continue reducing dependence on chat history.

---

# 27. Prefer Incremental Standardisation

The project repeatedly benefited from introducing standards gradually.

Examples include:

- Reading Experience;
- Hero image standards;
- Brand Guidelines;
- QA process;
- Development Standards.

Trying to define everything in advance would have produced weaker standards.

Instead, standards were documented after proving useful in practice.

This approach should continue.

Document proven patterns.

Do not document assumptions.

---

# 28. User Experience Includes the Team

One important insight is that maintainability affects user experience indirectly.

When the project is easier to maintain:

- guides stay up to date;
- improvements arrive faster;
- fewer inconsistencies appear;
- quality remains higher over time.

Therefore, reducing maintenance effort is not only a technical objective.

It is also an investment in future visitor experience.

---

# 29. Do Not Confuse Activity with Progress

Many possible tasks can create the impression of momentum while adding little strategic value.

Examples include:

- endless visual refinements;
- creating unnecessary components;
- introducing features before they are needed.

Real progress usually strengthens one of these areas:

- authority;
- trust;
- maintainability;
- editorial quality;
- discoverability.

Future sprint planning should measure progress against these outcomes rather than against the number of completed tasks.

---

# 30. Protect the Project's Identity

As the project grows, there will always be opportunities to imitate larger travel websites.

This should be resisted unless there is a clear strategic advantage.

Milan Red Line should never attempt to become a generic tourism portal.

Its identity comes from combining:

- carefully selected travel guidance;
- local expertise;
- practical planning;
- accommodation integrated naturally into the visitor journey.

Every new feature, guide or partnership should reinforce this identity rather than dilute it.

Growth should make the project more recognisable, not more generic.

A visitor should increasingly feel:

"This is exactly the kind of information I was looking for."

That feeling is one of Milan Red Line's strongest long-term competitive advantages.

## Part 4 — Future Governance & Strategic Discipline

Status: Living document

---

# 31. Scale the System, Not the Workload

One of the project's implicit objectives has gradually become clear.

Growth should not require proportionally more effort.

Whenever the project expands, ask:

"Can this still be maintained comfortably in two years?"

If the answer is no, the solution should probably be redesigned before expanding further.

Scalability is not measured only by traffic.

It is measured by how much additional maintenance each new guide, feature or apartment creates.

The preferred outcome is that future growth mostly involves publishing content rather than maintaining infrastructure.

---

# 32. Preserve Editorial Independence

Milan Red Line should always remain free to decide what deserves to be published.

Content should never be driven primarily by:

- affiliate opportunities;
- temporary trends;
- commercial partnerships;
- seasonal hype.

Commercial opportunities are welcome only when they naturally align with the editorial mission.

Readers should always feel that recommendations exist because they are useful, not because they are profitable.

Trust is significantly harder to rebuild than revenue is to replace.

---

# 33. Analytics Should Gradually Replace Intuition

The first phase of the project necessarily relied on experience and judgement.

Over time this balance should change.

As reliable Analytics data accumulates, future decisions should increasingly answer questions such as:

- Which guides genuinely help visitors?
- Which search intents deserve expansion?
- Where do readers stop reading?
- Which internal links are actually followed?
- Which pages introduce visitors to the website?

However, analytics should refine judgement rather than replace it.

The strongest decisions combine:

experience

+

evidence.

---

# 34. Preserve Architectural Stability

The project now has a mature architecture.

Future PMs should treat this as an asset.

Large structural changes should be exceptional rather than routine.

Whenever a proposal requires changing several core systems simultaneously, first ask:

"Can the same outcome be achieved by extending the current architecture?"

Most of the time the answer will be yes.

Evolution has repeatedly proved more efficient than redesign.

---

# 35. Every New System Needs an Owner

As Milan Red Line grows, new systems will appear.

Examples may include:

- multilingual workflows;
- analytics dashboards;
- databases;
- guest management;
- automation.

Every permanent system should have a clearly defined owner.

Without ownership, systems gradually become inconsistent.

Ownership does not necessarily imply a person.

It may also represent a primary responsibility area within the project.

---

# 36. Growth Should Increase Consistency

A common mistake in growing projects is allowing each new feature to introduce its own conventions.

Milan Red Line deliberately follows the opposite philosophy.

Each new feature should make the project feel more coherent.

Visitors should experience:

- similar navigation;
- similar page structures;
- similar visual language;
- similar writing style;
- similar interaction patterns.

Consistency creates confidence.

Confidence builds trust.

---

# 37. Resist Feature Inflation

One of the greatest long-term risks is gradually accumulating features that individually seem harmless.

Examples include:

- additional filters;
- optional settings;
- decorative animations;
- multiple navigation paths;
- duplicate interfaces.

Each feature adds:

- cognitive load;
- testing effort;
- documentation;
- maintenance.

Future PMs should regularly ask:

"If this feature disappeared tomorrow, would users genuinely miss it?"

If the answer is uncertain, the feature probably does not deserve implementation.

---

# 38. Continue Building Evergreen Assets

The project's strongest assets are those that continue generating value without constant intervention.

Examples include:

- evergreen guides;
- reusable components;
- documentation;
- Knowledge Network;
- Brand Guidelines.

These assets compound over time.

Temporary campaigns rarely do.

Whenever possible, prioritise investments that remain useful long after the sprint ends.

---

# 39. Protect the Pace of the Project

A sustainable pace has consistently produced better outcomes than bursts of activity.

Large periods of rushed implementation often increase the amount of future maintenance.

Instead, the project has benefited from a rhythm based on:

- planning;
- implementation;
- verification;
- documentation;
- handover.

Maintaining this cycle is more valuable than temporarily increasing output.

Long-term consistency is one of the project's competitive strengths.

---

# 40. Success Should Be Measured Broadly

Direct bookings remain the primary business objective.

However, they are not the only indicator of progress.

Healthy signals include:

- stronger topical authority;
- increasing Search Console impressions;
- improving click-through rates;
- richer Knowledge Network;
- decreasing maintenance effort;
- more reusable systems;
- higher documentation quality.

These indicators often improve before business metrics do.

Future PMs should recognise and value this sequence.

The project grows through accumulated quality rather than isolated milestones.

## Part 5 — Practical Heuristics & Final Principles

Status: Living document

---

# 41. Ask "Why?" Before "How?"

When a new idea emerges, resist discussing implementation immediately.

Instead ask:

- Why does this matter?
- What visitor problem does it solve?
- Why now?
- Why is it better than the next alternative?

Good implementation cannot compensate for solving the wrong problem.

---

# 42. Preserve Focus

The project should always have one dominant direction.

At any given time it should be possible to answer:

"What are we trying to improve this month?"

If that answer becomes unclear, priorities have probably become diluted.

Focus creates momentum.

Momentum creates progress.

---

# 43. Small Decisions Accumulate

Most improvements made to Milan Red Line were individually modest.

Examples include:

- refining a heading;
- simplifying a paragraph;
- improving metadata;
- updating Related Guides;
- creating one reusable component.

Taken individually they seem minor.

Taken together they transformed the project.

Future PMs should continue valuing cumulative improvement over occasional large redesigns.

---

# 44. Avoid Temporary Solutions Becoming Permanent

Sometimes a quick solution is justified.

However, temporary solutions have a habit of remaining in production indefinitely.

Whenever a compromise is accepted:

- document it;
- explain why it exists;
- decide whether it should eventually disappear.

A documented compromise is manageable.

An undocumented one becomes technical or editorial debt.

---

# 45. Every Improvement Should Leave a Trace

Whenever something valuable is learned, ask:

"Where should this knowledge live?"

Possible destinations include:

- Blueprint;
- Roadmap;
- Development Standards;
- Content Standards;
- Knowledge Network;
- Brand Guidelines;
- PM Continuity Notes;
- Development Continuity Notes.

The objective is to reduce reliance on memory over time.

Knowledge should gradually migrate from conversations into documentation.

---

# 46. Stability Is More Valuable Than Novelty

Visitors rarely notice every new feature.

They do notice when the experience feels reliable.

Consistency across:

- layout;
- navigation;
- writing;
- branding;
- interaction;

creates a stronger impression than frequent redesigns.

Future PMs should therefore value stability as a feature in itself.

---

# 47. Finish the Last 10%

The final part of a sprint often creates the greatest long-term value.

Examples include:

- updating documentation;
- reviewing internal links;
- validating metadata;
- checking mobile layouts;
- writing the handover.

These activities may feel less exciting than implementation, but they determine how maintainable the project remains.

A sprint should not be considered complete simply because the feature works.

It is complete when the project has fully absorbed the change.

---

# 48. Respect the Reader's Time

Every editorial and UX decision should acknowledge that visitors arrive with a goal.

Readers are usually trying to answer practical questions quickly.

Avoid unnecessary introductions.

Avoid repetition.

Avoid information that delays the answer.

The project earns trust by helping visitors reach useful conclusions with as little effort as possible.

---

# 49. The Project Is Built on Trust

Traffic can fluctuate.

Algorithms can change.

Search trends can evolve.

Trust is the most durable asset the project can create.

Trust comes from many small decisions:

- honest recommendations;
- realistic expectations;
- accurate information;
- transparent communication;
- practical guidance.

Whenever a decision risks increasing clicks at the expense of trust, trust should prevail.

The long-term strategy depends on it.

---

# 50. Final Principle

Milan Red Line should continue becoming easier to understand, easier to maintain and more useful every time it evolves.

The project should not aim to become the biggest travel website.

It should aim to become one of the most trusted resources for people planning a stay in Milan and its surrounding area.

Every future decision should ideally strengthen at least one of these three qualities:

- usefulness;
- trust;
- simplicity.

If a proposed improvement strengthens all three, it is almost certainly aligned with the project's long-term direction.

If it weakens one of them, the decision deserves careful reconsideration.

---

# Closing Note

This document intentionally captures judgement rather than procedures.

Procedures evolve.

Technologies change.

Documentation grows.

The reasoning behind good decisions should remain remarkably stable.

Future Project Managers are encouraged to evolve the project, but they should do so by extending its principles rather than replacing them.

The objective is continuity, not repetition.

A successful future version of Milan Red Line should still feel unmistakably like Milan Red Line.

















# Appendix A — Chat Lifecycle & Saturation Standard (CLSS)

**Status:** Stable

**Owner:** Project Management

---

# Purpose

This appendix defines the standard lifecycle conventions for every long-running Milan Red Line chat.

Its objectives are to:

- maintain fast and focused conversations;
- minimise dependence on chat history;
- encourage continuous migration of stable knowledge into permanent documentation;
- identify the right moment to create a successor chat.

These conventions apply to every specialised project chat, including Project Management, Development, Content & SEO, UX / Design, Analytics & Growth and any future specialised conversations.

---

# Chat Lifecycle Philosophy

Chats are working environments rather than permanent knowledge repositories.

Permanent knowledge should progressively migrate into project documentation.

The ideal lifecycle is therefore:

New Chat

↓

Work

↓

Documentation

↓

Handover

↓

Successor Chat

The project should become progressively less dependent on any individual conversation.

---

# Chat Lifecycle & Saturation Standard (CLSS)

The saturation score estimates how much contextual information the current conversation is carrying.

It does **not** measure:

- the overall maturity of the project;
- the amount of permanent documentation;
- the total number of project files.

Instead, it estimates how much accumulated conversational context increases reasoning complexity and gradually reduces responsiveness.

Knowledge that has already been migrated into permanent documentation reduces the effective saturation because future reasoning no longer depends exclusively on chat history.

The objective is consistency rather than mathematical precision.

---

# Saturation Scale

## 0–20 — New Chat

Characteristics:

- minimal context;
- extremely fast responses;
- requires onboarding.

Recommended action:

None.

---

## 20–40 — Young Chat

Characteristics:

- understands its role;
- fast responses;
- limited reasoning overhead.

Recommended action:

None.

---

## 40–60 — Mature Chat

Characteristics:

- strong contextual understanding;
- excellent balance between speed and continuity;
- ideal operating range.

Recommended action:

None.

---

## 60–75 — Rich Context

Characteristics:

- increasing accumulated knowledge;
- slightly slower reasoning;
- continuity becoming valuable.

Recommended action:

Begin considering a future handover.

---

## 75–85 — Heavy Chat

Characteristics:

- significant historical context;
- noticeable reasoning overhead;
- strong continuity value.

Recommended action:

Prepare a handover and plan a successor chat after the current major sprint.

---

## 85–100 — Saturated Chat

Characteristics:

- high contextual overhead;
- slower responses;
- diminishing returns from continuing the conversation.

Recommended action:

Create a successor chat as soon as practical.

---

# Evaluation Criteria

Every chat should estimate its saturation using five equally weighted factors.

Each contributes approximately 20% of the final score.

## 1. Accumulated Context

How much historical information has been retained?

---

## 2. Strategic Decisions

How many important decisions still exist primarily inside the conversation?

---

## 3. Topic Breadth

How many different subjects has the chat covered?

---

## 4. Context Dependencies

How often must previous discussions be reconstructed before answering?

---

## 5. Reasoning Overhead

How much accumulated conversational context increases processing complexity?

---

# Interpretation

The saturation score is intentionally approximate.

Consistency between chats is more important than exact numerical precision.

Differences of approximately ±3 points should be considered insignificant.

Every evaluation should also include a confidence level indicating how reliable the estimate is.

Confidence does not measure the quality of the chat.

It measures how confidently the evaluator can place the current conversation within a saturation range.

## Confidence Levels

### High

The conversation clearly belongs to a specific saturation range.

Small differences in interpretation would not change the recommended action.

Typical examples:

- 18
- 43
- 76
- 91

---

### Medium

The conversation is close to the boundary between two saturation ranges.

Minor differences in interpretation could slightly change the numerical score, although the overall recommendation normally remains the same.

Typical examples:

- 38–42
- 58–62
- 73–77
- 83–87

---

### Low

The available context is incomplete or unusually ambiguous.

Additional information about the conversation could significantly change the evaluation.

This level should be uncommon and usually occurs only when the evaluator has limited visibility of the conversation history.

---

# Standard Project Commands

The following commands are recognised as project conventions.

They provide a consistent way to manage long-running conversations.

---

## Command: valutachat

Purpose

Estimate the current saturation level of the active conversation.

The evaluation always refers to the current chat only.

It does not evaluate the overall project.

Expected output:

Chat:

<Name of the current chat>

Score:

<0–100>

Confidence:

High / Medium / Low

Assessment:

3–5 concise lines explaining the evaluation.

Recommendation:

- Continue normally
- Prepare handover
- Create successor chat

---

## Command: handover

Purpose

Generate a complete handover suitable for opening a successor chat.

The handover should include:

- current project status;
- completed work;
- important decisions not yet documented;
- immediate priorities;
- open discussions;
- recommended next sprint.

The handover should avoid repeating information already contained in permanent project documentation.

If necessary, it may be divided into multiple parts.

---

## Command: checkfonti

Purpose

Verify whether the permanent documentation adequately represents the knowledge accumulated inside the current conversation.

The review should identify:

- knowledge that should migrate into documentation;
- documentation requiring updates;
- duplicated information;
- obsolete information;
- opportunities to simplify future onboarding.

The objective is to minimise long-term dependence on chat history.

---

# Recommended Workflow

At the end of every major sprint:

1. Complete the implementation.
2. Update the relevant permanent documentation.
3. Run `checkfonti`.
4. Produce the `handover` if appropriate.
5. Run `valutachat`.
6. If the conversation reaches the Heavy or Saturated range, create a successor chat after the sprint concludes.

This workflow keeps project knowledge well documented while maintaining responsive and focused conversations.
