# 09_Search_and_Metadata

> **Module Owner:** Development / Content & SEO
> **Status:** Living document

---

# Purpose

This document defines the permanent standards governing search, metadata and technical SEO for Milan Red Line.

It also consolidates the remaining technical SEO guidance previously maintained in the legacy SEO Blueprint.

---

# Core Principle

Search optimisation exists to help users discover high-quality content.

Technical SEO supports useful content; it never replaces it.

---

# Metadata Standards

Every guide should include:

- title;
- meta description;
- keywords;
- destinations;
- search aliases;
- reading time;
- publication date;
- hero image metadata.

Metadata should accurately describe the page rather than maximise keyword density.

---

# URL Standards

URLs should be:

- short;
- descriptive;
- stable;
- lowercase;
- evergreen.

Avoid changing published URLs unless absolutely necessary.

---

# Canonical Strategy

Every published guide should expose the correct canonical URL.

Canonical URLs prevent duplication and preserve search equity when content evolves.

---

# Internal Search

Guide Search should index:

- title;
- description;
- keywords;
- destinations;
- search aliases.

Search results should prioritise relevance rather than publication date.

---

# Image Metadata

Every guide hero image should include:

- descriptive alt text;
- optimised WebP asset;
- meaningful filename.

Avoid generic names such as image1.webp.

---

# Technical SEO Checklist

Before publication verify:

- canonical URL;
- metadata complete;
- clean headings;
- internal links;
- image optimisation;
- accessibility basics;
- responsive layout;
- successful production build.

Future additions:

- structured data expansion;
- hreflang for multilingual support.

---

# Search Console

Search Console should be used to:

- identify indexing issues;
- monitor coverage;
- discover new search queries;
- identify opportunities for guide improvements.

---

# Analytics

Analytics supports editorial decisions but should not drive low-quality content production.

Meaningful measurements become part of project governance only when sufficient data is available.
